# Лабораторная работа 5: Моделирование Scrum

## Описание проекта
Консольное приложение для управления задачами. Пользователи могут создавать новые задачи, просматривать существующие, отмечать их как выполненные и удалять неактуальные. Все данные сохраняются в локальный файл. Проект предназначен для изучения Scrum методологии на практике.

## Product Backlog
Создано 8 задач (Issues) в репозитории. Каждая задача имеет четкий заголовок, развернутое описание функциональности и конкретные критерии готовности (definition of done) в виде маркированного списка требований.

## Sprint 1 Planning
Для первого спринта выбрано 4 ключевые задачи из беклога. Все они помечены меткой `sprint-1`. Для управления статусом добавлены метки `in-progress` (для начатых задач) и `todo` (для запланированных). В описании каждой задачи указаны: конкретная цель, ожидаемый результат к концу спринта, оценка сложности от 1 до 5 story points и приоритет выполнения (high, medium или low).

## Scrum-артефакты в Issues
Каждая задача спринта содержит полный набор артефактов: детальное описание что нужно сделать, definition of done с четкими критериями завершения, приоритет выполнения, оценку сложности в story points и явное указание принадлежности к спринту через метку `sprint-1`.

## Ход спринта
Прогресс смоделирован через изменение статусов задач:
- **2 задачи** переведены в состояние `in-progress` (начаты)
- **2 задачи** успешно завершены и закрыты (`closed`)
- **1 задача** оставлена незавершенной (остается в `todo`)

## Sprint Review
**Завершенные задачи:**
1. Создание базовой структуры проекта - все файлы и папки созданы
2. Реализация функции добавления задач - работает с сохранением в файл

**Доставленный инкремент:** Работающее консольное приложение, позволяющее пользователю создавать новые задачи и сохранять их в файл. Есть базовый интерфейс ввода.

**Незавершенные задачи:**
- Функция просмотра списка задач - частично реализована, но не полностью
- Функция удаления задач - не была начата из-за нехватки времени

## Sprint Retrospective
**Что прошло хорошо:** Четко сформулированные критерии готовности помогали понимать когда задача завершена. Простые задачи были правильно оценены по времени. Регулярное обновление статусов в Issues позволяло видеть общий прогресс.

**Что пошло не так:** Сложность работы с файловой системой оказалась выше ожидаемой. Не хватило времени на обработку ошибок и исключительных ситуаций. Планирование было слишком оптимистичным относительно возможностей команды.

**Что можно улучшить:** Более детально разбивать сложные задачи на подзадачи меньшего размера. Добавлять отдельные задачи по обработке ошибок и тестированию. Проводить короткие ежедневные встречи для отслеживания прогресса. Создавать прототипы функций перед их полной реализацией.
